# Restaurant POS System - Implementation Plan

## Tech Stack
- **Framework:** Angular 21 (standalone components)
- **Styling:** TailwindCSS v4 (already installed)
- **State:** Angular signals + RxJS for HTTP
- **Routing:** Angular Router with lazy loading
- **Auth:** JWT token stored in localStorage, `Authorization: Bearer <token>` header via HTTP interceptor

## 3. Implementation Phases

### Phase 1: Core Infrastructure

#### 3.1 Update `app.config.ts`
- Add `provideHttpClient(withInterceptors([authInterceptor]))` 
- Keep `provideRouter(routes)`

#### 3.2 Update Interfaces
- Update `register.interface.ts` to add `LoginResponse` interface:
  ```ts
  export interface LoginResponse {
    token: string;
    user: {
      id: number;
      username: string;
      role: 'Cashier' | 'Chef' | 'Admin';
    };
  }
  ```
- Update `dishes.interface.ts`: remove `imageUrl` field from `Dish` (backend doesn't handle images)
- Add `DishCreate` interface (for POST/PUT body without `id`):
  ```ts
  export interface DishPayload {
    name: string;
    price: number;
    categoryId: number;
    active?: boolean;
  }
  ```

#### 3.3 Auth Service (`auth.service.ts`)
- **Methods:**
  - `login(credentials: Login): Observable<LoginResponse>` — POST to `/api/auth/login`, store token + user in localStorage, set `currentUser` signal
  - `register(data: Register): Observable<any>` — POST to `/api/auth/register`
  - `logout()` — clear localStorage, reset signals, navigate to menu
  - `getToken(): string | null` — retrieve token from localStorage
  - `isLoggedIn(): boolean` — computed from currentUser signal
  - `currentUser: Signal<LoginUser | null>` — current user signal
  - `isBoss(): boolean` — computed, checks `currentUser().id === 1`
  - `hasRole(...roles: string[]): boolean` — check if current user has one of the given roles
- **Initialization:** On app start, check localStorage for existing token/user and hydrate the signal

#### 3.4 Auth Interceptor (`auth.interceptor.ts`)
- Functional interceptor (`HttpInterceptorFn`)
- Attach `Authorization: Bearer <token>` to every outgoing request if token exists

#### 3.5 Route Guards
- **`authGuard`**: Redirect to `/login` if not authenticated
- **`roleGuard`**: Factory function accepting allowed roles, redirects to `/menu` if role doesn't match
  - Usage: `canActivate: [authGuard, roleGuard('Admin')]`

#### 3.6 Implement All Services

**DishesService:**
- `getAll(): Observable<Dishes>` — GET `/api/dishes`
- `getById(id: number): Observable<Dish>` — GET `/api/dishes/{id}`
- `getByCategory(categoryId: number): Observable<Dishes>` — GET `/api/dishes?categoryId={categoryId}`
- `create(dish: DishPayload): Observable<any>` — POST `/api/dishes`
- `update(id: number, dish: DishPayload): Observable<any>` — PUT `/api/dishes/{id}`

**CategoriesService:**
- `getAll(): Observable<Categories>` — GET `/api/categories`
- `getById(id: number): Observable<Category>` — GET `/api/categories/{id}`
- `create(name: string): Observable<any>` — POST `/api/categories/{name}` (no body)
- `update(id: number, data: { NewName?: string; Active?: boolean }): Observable<any>` — PUT `/api/categories/{id}`

**OrdersService:**
- `getAll(): Observable<Orders>` — GET `/api/orders`
- `getById(id: number): Observable<Order>` — GET `/api/orders/{id}`
- `checkout(data: { items: OrderItem[]; customerName: string }): Observable<{ orderId: number; status: string }>` — POST `/api/orders/checkout`
- `updateStatus(id: number, status: 'Pending' | 'In Progress' | 'Completed' | 'Delivered'): Observable<any>` — PUT `/api/orders/{id}/status`
- `delete(id: number): Observable<any>` — DELETE `/api/orders/{id}`

**UsersService:**
- `getAll(): Observable<Users>` — GET `/api/users`
- `getById(id: number): Observable<User>` — GET `/api/users/{id}`
- `update(id: number, data: { username?: string; password?: string; role?: string }): Observable<any>` — PUT `/api/users/{id}`
- `delete(id: number): Observable<any>` — DELETE `/api/users/{id}`

### Phase 3: Pages / Features

#### 3.9 Menu Page (`/menu`) — PUBLIC (default route)
- **Purpose:** Display the restaurant menu, viewable by everyone
- **UI:**
  - Top: Category filter bar (horizontal scrollable pills/tabs) — only shows active categories
  - "All" tab selected by default
  - Grid of dish cards (responsive: 1 col mobile, 2 col tablet, 3-4 col desktop)
  - Each card shows: dish name, price, category badge
  - Only active dishes shown to non-admin users
- **Cashier enhancement:** Each dish card gets an "Add to Order" button when logged in as Cashier/Admin
  - Quantity selector (+ / -) appears on the card
  - Floating cart/order summary bar at the bottom showing item count and total
  - "Proceed to Checkout" button in the floating bar

#### 3.10 Login Page (`/login`)
- **UI:** Centered card with:
  - Username field
  - Password field  
  - Login button
  - Error message display on failed login
- **Behavior:** On success, redirect based on role:
  - Admin → `/admin`
  - Chef → `/kitchen`
  - Cashier → `/menu` (to start taking orders)

#### 3.11 Orders / Checkout Page (`/orders`) — Cashier & Admin
- **Guard:** `authGuard` + `roleGuard('Cashier', 'Admin')`
- **Purpose:** Take a new order and checkout
- **UI:**
  - This is the order-building flow from the menu
  - Cart/order summary showing all added items (dish name, quantity, notes input, subtotal per item)
  - Customer name input field
  - Individual item notes field
  - Running total
  - "Checkout" button
- **On successful checkout:**
  - Display a success modal/overlay showing the **Order ID** prominently (large font)
  - Message: "Show this Order ID to the customer"
  - "New Order" button to reset and start fresh

#### 3.12 Kitchen Page (`/kitchen`) — Chef, Cashier, Admin
- **Guard:** `authGuard` + `roleGuard('Chef', 'Cashier', 'Admin')`
- **Purpose:** View active orders (Pending, In Progress, Completed) and manage their status. Delivered orders are NOT shown here — they are only visible in the Admin panel's Manage Orders page.
- **UI:**
  - Three-column Kanban-style layout (or tab-based on mobile):
    - **Pending** column — new orders
    - **In Progress** column — orders being prepared
    - **Completed** column — finished orders, ready for pickup
  - Each order card shows:
    - Order ID
    - Customer name
    - Items list (dish name x quantity, notes if any)
    - Total
    - Timestamp
    - Status badge (color-coded)
  - **Chef/Admin actions on each card:**
    - Status change buttons (contextual):
      - Pending → "Start Preparing" (changes to In Progress)
      - In Progress → "Mark Completed"
      - Completed → "Mark Delivered" (changes to Delivered — card disappears from kitchen view)
    - Cancel button (only on Pending orders, for logged-in users)
  - **Cashier actions:**
    - Cancel button (only on Pending orders)
  - **Delivered orders are filtered out** — once an order is marked `Delivered`, it is removed from the kitchen display so the Chef has a clean view of only active orders
  - Auto-refresh: Poll every 15 seconds or manual refresh button

#### 3.13 Admin Panel (`/admin`) — Admin only
- **Guard:** `authGuard` + `roleGuard('Admin')`
- **Layout:** Sidebar navigation (or top tabs on mobile) with sections:

##### 3.13.1 Admin Dashboard (`/admin`)
- Quick overview: total orders today, active dishes count, active categories count, user count
- Quick links to each management section

##### 3.13.2 Manage Dishes (`/admin/dishes`)
- **Table/list view** showing all dishes (active & inactive)
- **Filters:** 
  - Toggle: Active / Inactive / All
  - Filter by category dropdown
- **Each row shows:** Name, Price, Category, Active status toggle
- **Actions per row:**
  - Edit (opens inline form or modal) — name, price, categoryId
  - Deactivate/Activate — PUT with `active: false/true`
- **"Add New Dish" button** — opens a form (modal or inline) with: name, price, category dropdown
- Deactivating = setting `active: false` via PUT (not DELETE)

##### 3.13.3 Manage Categories (`/admin/categories`)
- **Table/list view** showing all categories (active & inactive)
- **Filter toggle:** Active / Inactive / All
- **Each row shows:** Name, Active status
- **Actions per row:**
  - Edit name — PUT with `{ NewName: "..." }`
  - Deactivate/Activate — PUT with `{ Active: false/true }`
  - **Warning on deactivate:** "Deactivating this category will also deactivate all dishes in this category"
- **"Add New Category" button** — simple name input, POST to `/api/categories/{name}`

##### 3.13.4 Manage Users (`/admin/users`)
- **Table/list view** showing all users
- **Each row shows:** ID, Username, Role, Actions
- **Role-based restrictions (enforced in UI):**
  - **Normal Admins (id !== 1):**
    - Can see and manage Cashier and Chef accounts
    - Can create Cashier and Chef accounts (via register endpoint)
    - Can edit username/password of Cashier/Chef accounts
    - Can delete Cashier/Chef accounts
    - CANNOT see/edit/delete Admin accounts
  - **Boss Admin (id === 1):**
    - Can see and manage ALL accounts (Cashier, Chef, Admin)
    - Can create Admin accounts too
    - Can edit other Admin accounts (username/password)
    - Can delete other Admin accounts
    - Cannot delete own account (id=1), but CAN edit own details
- **"Register New User" button** — opens form with: username, password, role dropdown
  - Role dropdown options depend on who is logged in (Boss sees Admin option, normal Admin doesn't)
- **Edit user modal/form:** username, password fields (role shown but not editable via this form)

##### 3.13.5 Manage Orders (`/admin/orders`)
- Full orders table with all orders (including Delivered)
- Filters: by status (Pending, In Progress, Completed, Delivered, All)
- Cancel (DELETE) pending orders
- Change status of any order
- Delivered orders are visible here (unlike the kitchen page) — this is where Admins can review order history including completed deliveries

## 5. Key Business Rules (enforced in frontend)

| Rule | Implementation |
|------|---------------|
| Roles are case-sensitive: `Admin`, `Chef`, `Cashier` | Use exact strings in all comparisons |
| Order statuses are case-sensitive: `Pending`, `In Progress`, `Completed`, `Delivered` | Use exact strings |
| Boss Admin has `id === 1` | Check `currentUser().id === 1` for super privileges |
| Normal Admins cannot manage Admin accounts | Filter out Admin users from the list unless Boss |
| Deactivating = `active: false` via PUT (not DELETE) | Use update endpoint, never delete endpoint for dishes/categories |
| Deactivating category auto-deactivates its dishes (backend) | Show warning in UI, refresh dishes list after category deactivation |
| Only Pending orders can be cancelled (DELETE) | Conditionally show cancel button only when `status === 'Pending'` |
| Customers (not logged in) can only view menu | No auth guard on `/menu`, hide all action buttons |
| Boss admin (id=1) cannot delete own account | Disable delete button for own account |
| Boss admin (id=1) CAN edit own account details | Allow edit form for self |
| Cash-only orders, display order ID on screen | Show large order ID in success modal after checkout |
| Delivered orders hidden from Kitchen page | Kitchen only shows Pending/In Progress/Completed; Delivered orders visible only in Admin > Manage Orders |
| Order lifecycle: Pending → In Progress → Completed → Delivered | Chef uses "Mark Delivered" on Completed orders when customer picks up food |
